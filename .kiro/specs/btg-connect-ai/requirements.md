# Requirements Document

## Introduction

BTG ConnectAI is an AI-powered conversational banking solution (Agentic AI) for BTG Pactual bank. The system provides an intelligent conversational assistant integrated with WhatsApp that understands natural language requests, maintains conversational context, and executes operational flows for financial queries and transactional request generation. The solution reduces friction for ~9,176 active clients and ~35 commercial staff by automating repetitive interactions currently handled manually by Relationship Managers (RMs) and Daily Bankers (DBs). The architecture is AWS-native, serverless-first, following the AWS Well-Architected Framework principles for security, reliability, performance efficiency, cost optimization, operational excellence, and sustainability.

## Glossary

- **Conversational_Agent**: The core Agentic AI system that interprets user messages, maintains context, coordinates tools, and orchestrates workflows to fulfill banking requests
- **WhatsApp_Gateway**: The integration layer that connects WhatsApp Business API with the Conversational_Agent via AWS services
- **NLU_Engine**: The Natural Language Understanding component powered by Amazon Bedrock that interprets user intent, extracts entities, and classifies requests
- **Context_Manager**: The component responsible for maintaining conversational memory and session state across interactions
- **Tool_Orchestrator**: The component that coordinates execution of different tools (queries, operations, validations) based on the Conversational_Agent's decisions
- **Financial_Query_Service**: The service layer that retrieves account movements, balances, product status, and spending analysis from bank core systems
- **Operation_Request_Service**: The service that generates operational orders such as transfers and operational requests through the bank's transactional APIs
- **Secure_Link_Generator**: The component that creates time-limited, cryptographically signed URLs redirecting users to official bank portals for authentication and critical validations
- **Traceability_Service**: The observability and audit component that records all AI decisions, tool invocations, and operational actions for governance and compliance
- **Bank_Client**: An active client of BTG Pactual who interacts with the system through WhatsApp
- **Session**: A bounded conversational interaction between a Bank_Client and the Conversational_Agent with maintained context
- **Operational_Request**: A formal request generated by the system (transfers, payments, operational orders) that requires validation through secure channels
- **Fallback_Response_Engine**: The component that provides template-based responses for common queries when the NLU_Engine (Amazon Bedrock) is unavailable
- **Fast_Path_Router**: The component that identifies simple, well-known queries and routes them directly to the appropriate service, bypassing the full orchestration chain
- **Idempotency_Key**: A unique identifier associated with a financial operation request that ensures the operation is processed exactly once regardless of duplicate submissions
- **Message_Deduplication_Service**: The component responsible for detecting and discarding duplicate inbound WhatsApp messages before they reach the Conversational_Agent
- **Bank_Core_API**: The external bank core system APIs that provide account data, transaction processing, and product information

## Requirements

### Requirement 1: WhatsApp Channel Integration

**User Story:** As a Bank_Client, I want to interact with my bank through WhatsApp, so that I can access banking services through a familiar and convenient messaging platform.

#### Acceptance Criteria

1. WHEN a Bank_Client sends a message via WhatsApp, THE WhatsApp_Gateway SHALL receive the message and forward it to the Conversational_Agent within 3 seconds
2. WHEN the Conversational_Agent produces a response, THE WhatsApp_Gateway SHALL deliver the response to the Bank_Client via WhatsApp within 2 seconds
3. THE WhatsApp_Gateway SHALL support text messages, interactive buttons, and list messages as response formats
4. WHEN the WhatsApp_Gateway receives a message from an unregistered phone number, THE WhatsApp_Gateway SHALL respond with a welcome message and identity verification prompt
5. IF the WhatsApp_Gateway loses connectivity to the WhatsApp Business API, THEN THE WhatsApp_Gateway SHALL queue outbound messages and retry delivery with exponential backoff for up to 5 minutes
6. THE WhatsApp_Gateway SHALL process concurrent messages from up to 10,000 Bank_Clients without degradation in response time

### Requirement 2: Natural Language Understanding

**User Story:** As a Bank_Client, I want to communicate with the assistant using natural language in Spanish, so that I can express my banking needs conversationally without learning specific commands.

#### Acceptance Criteria

1. WHEN a Bank_Client sends a message in natural language (Spanish), THE NLU_Engine SHALL identify the user intent with a confidence score
2. WHEN the NLU_Engine identifies an intent with confidence above 0.85, THE Conversational_Agent SHALL proceed with the identified intent
3. WHEN the NLU_Engine identifies an intent with confidence between 0.5 and 0.85, THE Conversational_Agent SHALL ask the Bank_Client a clarifying question
4. WHEN the NLU_Engine identifies an intent with confidence below 0.5, THE Conversational_Agent SHALL inform the Bank_Client that the request was not understood and suggest available options
5. THE NLU_Engine SHALL extract financial entities including account numbers, amounts, dates, currency types, and product names from user messages
6. THE NLU_Engine SHALL handle colloquial Spanish expressions, abbreviations, and common banking terminology used in the Colombian market
7. WHEN a Bank_Client sends a message containing multiple intents, THE NLU_Engine SHALL identify all intents and THE Conversational_Agent SHALL address them sequentially

### Requirement 3: Conversational Context Management

**User Story:** As a Bank_Client, I want the assistant to remember what we discussed earlier in the conversation, so that I do not have to repeat information in every message.

#### Acceptance Criteria

1. THE Context_Manager SHALL maintain conversational context for the duration of a Session
2. WHEN a Bank_Client references a previous topic within the same Session, THE Context_Manager SHALL resolve the reference using stored conversation history
3. THE Context_Manager SHALL store a maximum of 50 conversation turns per Session
4. WHEN a Session has been inactive for 30 minutes, THE Context_Manager SHALL expire the Session and notify the Bank_Client that a new Session will begin
5. WHEN a new Session begins, THE Context_Manager SHALL retain the Bank_Client identity but clear conversational context from the previous Session
6. THE Context_Manager SHALL resolve pronoun references and implicit entities using conversation history within the current Session

### Requirement 4: Account Movement and Transaction Queries

**User Story:** As a Bank_Client, I want to ask about my account movements and transactions, so that I can monitor my financial activity without calling my banker.

#### Acceptance Criteria

1. WHEN a Bank_Client requests account movements, THE Financial_Query_Service SHALL retrieve transactions for the specified account and date range from the bank core system
2. WHEN a Bank_Client requests account movements without specifying a date range, THE Financial_Query_Service SHALL return transactions from the last 30 days
3. THE Financial_Query_Service SHALL present transaction data including date, description, amount, and running balance in a readable conversational format
4. WHEN the Financial_Query_Service retrieves more than 10 transactions, THE Conversational_Agent SHALL summarize the results and offer pagination or filtering options
5. IF the Financial_Query_Service cannot retrieve data from the bank core system, THEN THE Conversational_Agent SHALL inform the Bank_Client of the temporary unavailability and suggest retrying later
6. WHEN a Bank_Client requests the current balance, THE Financial_Query_Service SHALL retrieve and display the available balance and total balance for the specified account

### Requirement 5: Spending Analysis and Behavior Statistics

**User Story:** As a Bank_Client, I want to receive analysis of my spending patterns, so that I can understand my financial behavior and make informed decisions.

#### Acceptance Criteria

1. WHEN a Bank_Client requests spending analysis, THE Financial_Query_Service SHALL categorize transactions and present spending distribution by category for the requested period
2. WHEN a Bank_Client requests spending analysis without specifying a period, THE Financial_Query_Service SHALL analyze the current calendar month
3. THE Financial_Query_Service SHALL identify the top 5 spending categories and present them with amounts and percentages
4. WHEN a Bank_Client asks about spending trends, THE Financial_Query_Service SHALL compare the current period with the previous equivalent period and highlight significant changes above 20%
5. THE Conversational_Agent SHALL present spending analysis in a clear conversational format with optional summary statistics

### Requirement 6: Product Status Inquiries

**User Story:** As a Bank_Client, I want to check the status of my financial products, so that I can stay informed about my investments, loans, and accounts without contacting my banker.

#### Acceptance Criteria

1. WHEN a Bank_Client asks about product status, THE Financial_Query_Service SHALL retrieve current status information for the specified product from the bank core system
2. THE Financial_Query_Service SHALL support status queries for savings accounts, checking accounts, investment portfolios, credit products, and certificates of deposit
3. WHEN a Bank_Client asks about product status without specifying a product, THE Conversational_Agent SHALL present a list of the Bank_Client's active products and ask which one to query
4. THE Financial_Query_Service SHALL present product information including current value, status, maturity date (where applicable), and interest rate
5. IF a product status query returns data older than 5 minutes, THEN THE Conversational_Agent SHALL indicate the timestamp of the last update to the Bank_Client

### Requirement 7: Operational Request Generation

**User Story:** As a Bank_Client, I want to initiate operational requests such as transfers through the conversational assistant, so that I can start banking operations without navigating complex interfaces.

#### Acceptance Criteria

1. WHEN a Bank_Client requests a transfer or operational order, THE Operation_Request_Service SHALL collect required parameters (destination account, amount, currency, description) through conversational interaction
2. WHEN all required parameters for an operational request are collected, THE Conversational_Agent SHALL present a summary of the operation and ask for confirmation
3. WHEN the Bank_Client confirms an operational request, THE Operation_Request_Service SHALL generate the formal Operational_Request and submit it to the bank's transactional system
4. THE Operation_Request_Service SHALL validate that requested amounts do not exceed the Bank_Client's available balance before generating the Operational_Request
5. WHEN an Operational_Request is successfully submitted, THE Conversational_Agent SHALL provide the Bank_Client with a confirmation number and expected processing time
6. IF the Operation_Request_Service fails to submit an Operational_Request, THEN THE Conversational_Agent SHALL inform the Bank_Client of the failure reason and suggest alternative actions

### Requirement 8: Secure Link Generation for Authentication

**User Story:** As a Bank_Client, I want to be redirected to secure bank portals when operations require formal authentication, so that my critical validations remain protected within authorized channels.

#### Acceptance Criteria

1. WHEN an operation requires authentication or formal validation, THE Secure_Link_Generator SHALL create a unique, cryptographically signed URL pointing to the official bank portal
2. THE Secure_Link_Generator SHALL include the operation context (type, parameters, session reference) encoded in the secure link so the bank portal can pre-populate the validation form
3. THE Secure_Link_Generator SHALL set an expiration time of 10 minutes on each generated link
4. WHEN a secure link expires without being used, THE Secure_Link_Generator SHALL invalidate the link and THE Conversational_Agent SHALL inform the Bank_Client that a new link can be generated
5. THE Secure_Link_Generator SHALL use HMAC-SHA256 signing with rotating keys to prevent link tampering
6. WHEN a Bank_Client clicks a secure link, THE bank portal SHALL validate the signature and expiration before presenting the authentication form

### Requirement 9: Agentic Workflow Orchestration

**User Story:** As a Bank_Client, I want the assistant to autonomously coordinate complex multi-step requests, so that I can accomplish banking tasks through a single natural conversation without managing each step manually.

#### Acceptance Criteria

1. WHEN a Bank_Client makes a complex request requiring multiple steps, THE Tool_Orchestrator SHALL decompose the request into an ordered sequence of tool invocations
2. THE Tool_Orchestrator SHALL execute tool invocations sequentially, passing outputs from one step as inputs to the next step
3. WHEN a tool invocation fails during a multi-step workflow, THE Tool_Orchestrator SHALL inform the Bank_Client of the failure point and offer to retry or abort the remaining steps
4. THE Conversational_Agent SHALL inform the Bank_Client of progress during multi-step workflows by providing status updates after each completed step
5. THE Tool_Orchestrator SHALL enforce a maximum of 10 tool invocations per single user request to prevent infinite loops
6. WHEN the Tool_Orchestrator determines that a workflow requires authentication, THE Tool_Orchestrator SHALL pause the workflow, trigger secure link generation, and resume upon successful validation

### Requirement 10: Operational Traceability and Audit

**User Story:** As a compliance officer, I want all AI decisions and operations to be fully traceable, so that the bank can audit interactions and demonstrate regulatory compliance.

#### Acceptance Criteria

1. THE Traceability_Service SHALL record every interaction between the Bank_Client and the Conversational_Agent including timestamps, message content, identified intents, and confidence scores
2. THE Traceability_Service SHALL record every tool invocation including tool name, input parameters, output results, execution duration, and success/failure status
3. THE Traceability_Service SHALL record every Operational_Request generated including request parameters, Bank_Client identity, approval status, and completion status
4. THE Traceability_Service SHALL retain audit records for a minimum of 7 years in compliance with Colombian financial regulations
5. WHEN a compliance officer queries audit records, THE Traceability_Service SHALL support filtering by Bank_Client identity, date range, operation type, and session identifier
6. THE Traceability_Service SHALL generate a unique correlation identifier for each Session that links all related audit records

### Requirement 11: AI Governance and Safety

**User Story:** As a risk manager, I want the AI system to operate within defined guardrails, so that the bank maintains control over AI-generated responses and prevents harmful or non-compliant outputs.

#### Acceptance Criteria

1. THE Conversational_Agent SHALL restrict responses to banking and financial topics relevant to BTG Pactual services
2. WHEN a Bank_Client asks about topics outside the banking domain, THE Conversational_Agent SHALL politely decline and redirect the conversation to available banking services
3. THE Conversational_Agent SHALL apply content filtering to prevent generation of financial advice, investment recommendations, or forward-looking statements about market performance
4. THE NLU_Engine SHALL detect and flag potential social engineering or fraud attempts in Bank_Client messages
5. WHEN the NLU_Engine detects a potential fraud pattern, THE Conversational_Agent SHALL terminate the interaction and THE Traceability_Service SHALL generate a security alert
6. THE Conversational_Agent SHALL include a disclaimer in responses involving financial data stating that information is for reference purposes and official records are available through bank portals

### Requirement 12: Scalability and Availability

**User Story:** As a platform engineer, I want the system to handle peak loads and maintain high availability, so that Bank_Clients receive consistent service quality during demand spikes.

#### Acceptance Criteria

1. THE Conversational_Agent SHALL maintain response times below 5 seconds for 95th percentile of requests under peak load conditions
2. THE WhatsApp_Gateway SHALL scale automatically to handle traffic spikes of up to 3x the average daily volume during month-end closing periods
3. THE Conversational_Agent SHALL maintain 99.9% availability measured on a monthly basis
4. IF a component of the system becomes unavailable, THEN THE Conversational_Agent SHALL degrade gracefully by disabling affected capabilities while maintaining basic conversational functionality
5. THE system SHALL deploy across multiple AWS Availability Zones to ensure resilience against single-zone failures
6. WHEN system load exceeds 80% of provisioned capacity, THE system SHALL trigger automatic scaling within 60 seconds

### Requirement 13: Observability and Monitoring

**User Story:** As a platform engineer, I want comprehensive observability into system behavior, so that I can detect issues proactively and maintain system health.

#### Acceptance Criteria

1. THE system SHALL emit structured logs for every request including request identifier, latency, status code, and component name to Amazon CloudWatch Logs
2. THE system SHALL publish custom metrics including message throughput, response latency percentiles, error rates, and active session count to Amazon CloudWatch Metrics
3. THE system SHALL generate distributed traces for every request spanning all service components using AWS X-Ray
4. WHEN error rate exceeds 5% over a 5-minute window, THE system SHALL trigger an alarm notification to the operations team
5. WHEN average response latency exceeds 5 seconds over a 5-minute window, THE system SHALL trigger an alarm notification to the operations team
6. THE system SHALL provide a real-time operational dashboard displaying key health metrics, active sessions, and error trends

### Requirement 14: Security and Data Protection

**User Story:** As a security architect, I want all data to be protected in transit and at rest with proper access controls, so that Bank_Client information remains confidential and the system meets regulatory requirements.

#### Acceptance Criteria

1. THE system SHALL encrypt all data at rest using AWS KMS managed keys with AES-256 encryption
2. THE system SHALL encrypt all data in transit using TLS 1.2 or higher
3. THE system SHALL authenticate all API calls between internal components using IAM roles with least-privilege policies
4. THE system SHALL mask sensitive financial data (account numbers, balances) in logs and traces, retaining only the last 4 digits of account numbers
5. WHEN a Bank_Client identity is verified, THE system SHALL issue a session token with a maximum validity of 30 minutes
6. THE system SHALL store no Bank_Client credentials and delegate all authentication to the bank's identity provider through secure link redirects
7. THE system SHALL comply with Colombian data protection regulations (Ley 1581 de 2012) regarding personal data handling and storage


### Requirement 15: Bedrock Fallback Mode

**User Story:** As a Bank_Client, I want to receive responses for common simple queries even when the AI engine is unavailable, so that I can still access basic banking information during service degradation.

#### Acceptance Criteria

1. WHEN the NLU_Engine (Amazon Bedrock) becomes unavailable, THE Fallback_Response_Engine SHALL activate and handle inbound messages using pre-defined response templates
2. WHILE the NLU_Engine is unavailable, THE Fallback_Response_Engine SHALL provide template-based responses for balance inquiries, last transactions queries, and product status checks
3. WHILE the NLU_Engine is unavailable, THE Fallback_Response_Engine SHALL inform the Bank_Client that the system is operating in limited mode and that complex requests cannot be processed
4. WHEN the NLU_Engine becomes available again, THE Fallback_Response_Engine SHALL deactivate and THE Conversational_Agent SHALL resume full agentic processing within 30 seconds
5. THE Fallback_Response_Engine SHALL maintain a library of at least 20 response templates covering the most frequent simple query types identified from historical usage data
6. THE system SHALL perform health checks on the NLU_Engine every 10 seconds to detect unavailability and trigger fallback activation within 30 seconds of failure detection

### Requirement 16: Fast Path for Simple Queries

**User Story:** As a Bank_Client, I want immediate responses to simple queries like balance checks, so that I do not experience unnecessary latency for straightforward requests that do not require complex reasoning.

#### Acceptance Criteria

1. WHEN a Bank_Client sends a message matching a well-known simple query pattern (balance check, last N transactions), THE Fast_Path_Router SHALL route the request directly to the Financial_Query_Service bypassing the full orchestration chain
2. THE Fast_Path_Router SHALL identify simple query patterns using lightweight keyword matching and pattern recognition without invoking the NLU_Engine
3. WHEN the Fast_Path_Router routes a request through the fast path, THE system SHALL deliver the response to the Bank_Client within 2 seconds end-to-end
4. WHEN the Fast_Path_Router cannot confidently classify a message as a simple query (confidence below 0.9), THE Fast_Path_Router SHALL route the message through the standard orchestration chain
5. THE Fast_Path_Router SHALL support configurable query patterns that can be added or modified without code deployment
6. THE Traceability_Service SHALL record fast path invocations with the same audit detail as standard orchestration path requests

### Requirement 17: Idempotency for Financial Operations

**User Story:** As a Bank_Client, I want my financial operations to be processed exactly once even if technical retries occur, so that I am not charged or debited multiple times for the same transaction.

#### Acceptance Criteria

1. WHEN a Bank_Client confirms an operational request, THE Operation_Request_Service SHALL generate a unique Idempotency_Key for the operation before submission
2. WHEN the Operation_Request_Service receives a duplicate submission with an existing Idempotency_Key, THE Operation_Request_Service SHALL return the result of the original operation without executing the operation again
3. THE Operation_Request_Service SHALL store Idempotency_Keys with their associated operation results for a minimum of 24 hours
4. WHEN a message is delivered multiple times due to at-least-once delivery semantics, THE Operation_Request_Service SHALL detect the duplicate using the Idempotency_Key and prevent double processing
5. THE Operation_Request_Service SHALL include the Idempotency_Key in the Operational_Request confirmation provided to the Bank_Client
6. THE Traceability_Service SHALL record all duplicate submission attempts including the original Idempotency_Key and timestamp of the duplicate

### Requirement 18: Lambda Cold Start Mitigation

**User Story:** As a Bank_Client, I want consistent response times regardless of traffic patterns, so that I do not experience delays when the system has been idle.

#### Acceptance Criteria

1. THE WhatsApp_Gateway Lambda function SHALL use provisioned concurrency with a minimum of 5 warm instances at all times
2. THE Conversational_Agent Lambda function SHALL use provisioned concurrency with a minimum of 5 warm instances at all times
3. WHEN traffic volume drops below the provisioned concurrency threshold during low-traffic periods (midnight to 6 AM local time), THE system SHALL maintain provisioned instances to ensure sub-second cold start response
4. THE system SHALL monitor cold start occurrences and WHEN cold start rate exceeds 5% of total invocations over a 15-minute window, THE system SHALL trigger an alarm notification to the operations team
5. WHEN traffic patterns indicate an upcoming peak period (based on historical data), THE system SHALL pre-scale provisioned concurrency 15 minutes before the anticipated peak

### Requirement 19: Configurable Tool Invocation Limits

**User Story:** As a platform engineer, I want the tool invocation limit to be configurable and sufficient for complex workflows, so that legitimate multi-step operations are not prematurely terminated.

#### Acceptance Criteria

1. THE Tool_Orchestrator SHALL enforce a configurable maximum number of tool invocations per single user request, with a default value of 10
2. THE Tool_Orchestrator SHALL allow the maximum tool invocation limit to be modified through configuration without code deployment
3. WHEN a complex workflow requires internal orchestration steps (validation, audit logging, context resolution), THE Tool_Orchestrator SHALL classify these as internal steps that do not count against the Bank_Client-facing tool invocation limit
4. WHEN the Tool_Orchestrator reaches 80% of the configured tool invocation limit during a workflow, THE Conversational_Agent SHALL inform the Bank_Client that the request is complex and may require simplification
5. THE system SHALL log and analyze tool invocation counts per workflow type to identify patterns where the configured limit is insufficient
6. WHERE a workflow type consistently requires more than 80% of the configured limit, THE system SHALL generate a recommendation alert for the operations team to review the limit configuration

### Requirement 20: Bank Core API Integration Resilience

**User Story:** As a platform engineer, I want resilient integration with bank core systems, so that the conversational assistant handles API failures, rate limits, and version changes gracefully without impacting Bank_Client experience.

#### Acceptance Criteria

1. THE system SHALL authenticate all requests to the Bank_Core_API using mutual TLS (mTLS) or OAuth2 client credentials flow with token rotation
2. WHEN the Bank_Core_API returns a rate limit response (HTTP 429), THE system SHALL implement exponential backoff with jitter and retry the request up to 3 times before reporting unavailability to the Bank_Client
3. THE system SHALL support API version negotiation with the Bank_Core_API by including version headers in requests and handling version deprecation responses gracefully
4. WHEN the Bank_Core_API introduces a new version, THE system SHALL support concurrent communication with both the current and previous API version for a minimum transition period of 30 days
5. THE system SHALL implement circuit breaker patterns for Bank_Core_API calls, opening the circuit after 5 consecutive failures and attempting recovery after 30 seconds
6. IF the Bank_Core_API is unavailable for more than 60 seconds, THEN THE Conversational_Agent SHALL inform the Bank_Client that banking data is temporarily unavailable and offer to notify them when service is restored
7. THE system SHALL cache non-sensitive, non-transactional Bank_Core_API responses (product catalogs, exchange rates) for up to 5 minutes to reduce API call volume

### Requirement 21: SQS Message Deduplication

**User Story:** As a Bank_Client, I want my messages to be processed exactly once, so that I do not receive duplicate responses or trigger duplicate operations from a single message.

#### Acceptance Criteria

1. THE Message_Deduplication_Service SHALL assign a unique message identifier to each inbound WhatsApp message upon receipt
2. WHEN the Message_Deduplication_Service detects a duplicate message (same message identifier received within a 5-minute window), THE Message_Deduplication_Service SHALL discard the duplicate and return the response from the original processing
3. THE WhatsApp_Gateway SHALL use SQS FIFO queues with message deduplication enabled for inbound message processing to guarantee exactly-once delivery to the Conversational_Agent
4. THE Message_Deduplication_Service SHALL use the WhatsApp message ID as the primary deduplication key, combined with a content hash as a secondary verification
5. THE Message_Deduplication_Service SHALL retain deduplication records for a minimum of 5 minutes to cover the SQS FIFO deduplication window
6. THE Traceability_Service SHALL record all detected duplicate messages including the original message identifier, duplicate count, and source timestamp
